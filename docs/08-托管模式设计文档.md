# 08 — 托管模式设计文档

> 工作流的全自动执行模式：用户提交创意 + 风格参数后，系统自主完成「剧本 → 角色 → 分镜 → 视频」全链路，无需逐步点击。

---

## 一、背景与目标

### 1.1 问题

当前手动模式下，用户需要：
1. 新建项目、输入剧本 → 等待 AI 解析
2. 审核剧本 → 点击生成角色三视图/场景六视图 → 逐张审核
3. 点击生成分镜图 → 逐镜审核
4. 点击生成视频 → 等待轮询 → 下载

全流程约 **30 分钟**，需 **10+ 次人工干预**。对于「快速出片验证创意」的场景效率不足。

### 1.2 目标

提供**托管模式**（Autonomous Mode），用户一次提交即可：

| 指标 | 手动模式 | 托管模式 |
|------|---------|---------|
| 人工干预次数 | 10+ | 0（或 1 次最终审核） |
| 端到端耗时 | ~30 min | ~8-15 min |
| 质量保障 | 主观判断 | 自动评分 + 质量门控 |
| 失败恢复 | 手动重试 | 自动 REFINE/PIVOT |
| 可中断恢复 | 不支持 | 检查点断点续传 |

### 1.3 设计原则

参考 `HarnessEngineeringMaster/` 框架的核心模式：

- **Plan → Build → Evaluate 循环**：每个阶段产出物经评估后决定推进/重试
- **REFINE vs PIVOT**：评分趋势上升则精化，停滞或下降则换方案
- **Sprint 契约**：每阶段预定义验收标准和加权评分
- **检查点恢复**：每次评估后自动保存状态，断电/崩溃后可继续
- **渐进式技能加载**：按需加载引擎模块，降低启动开销

## 三、四阶段管线

### 3.1 阶段定义

| 阶段 | 引擎 | 生成模型 | 输出产物 | 通过门槛 | 关键性 |
|------|------|---------|---------|---------|--------|
| `script` | ScriptEngine | claude-sonnet-4-6 (LLM) | JSON 剧本结构 | 7.0 | 关键（失败终止） |
| `characters` | CharacterFactory | nano-banana-2 (图像) | 9 张图/角色 | 7.0 | 非关键（best effort） |
| `storyboard` | StoryboardGenerator | nano-banana-2 (图像) | 分镜 PNG | 7.0 | 非关键（best effort） |
| `video` | VideoComposer | doubao-seedance-2.0 (视频) | MP4 视频 | 6.0 | 关键（失败终止） |

### 3.2 单阶段执行流程

```
                    ┌──────────────┐
                    │ 开始阶段      │
                    └──────┬───────┘
                           ▼
              ┌─ attempt = 1..max_attempts ─┐
              │                             │
              │  ┌───────────────────────┐  │
              │  │ ① 策略决策            │  │
              │  │   首次: None          │  │
              │  │   后续: REFINE/PIVOT  │  │
              │  └───────────┬───────────┘  │
              │              ▼              │
              │  ┌───────────────────────┐  │
              │  │ ② BUILD: 调用引擎     │  │
              │  │   返回 artifact_id    │  │
              │  └───────────┬───────────┘  │
              │              ▼              │
              │  ┌───────────────────────┐  │
              │  │ ③ WAIT: 轮询完成      │  │
              │  │   间隔 5s, 超时 600s  │  │
              │  └───────────┬───────────┘  │
              │              ▼              │
              │  ┌───────────────────────┐  │
              │  │ ④ EVALUATE: 质量评估   │  │
              │  │   结构检查 + LLM 评分  │  │
              │  └───────────┬───────────┘  │
              │              ▼              │
              │  ┌───────────────────────┐  │
              │  │ ⑤ CHECKPOINT: 保存    │  │
              │  └───────────┬───────────┘  │
              │              ▼              │
              │     score >= threshold?     │
              │        ├─ YES → PASSED ─────┼──► 推进到下一阶段
              │        └─ NO → 下一轮 ──────┤
              └─────────────────────────────┘
                           │
                     超出 max_attempts
                           ▼
                  关键阶段? → 管线终止
                  非关键?   → 警告，继续
```

### 3.3 REFINE vs PIVOT 决策

REFINE/PIVOT 由 `WorkflowOrchestrator._decide_strategy()` 根据**分数趋势**决定（不是由评估器返回）：

```python
delta = score_history[-1] - score_history[-2]

if delta > 0.5:   → "REFINE"  # 大幅进步，保持方向
if 0 < delta:     → "REFINE"  # 有进展，继续精化
if delta == 0:    → "PIVOT"   # 停滞，换方案
if delta < 0:     → "PIVOT"   # 退步，必须换方案
```

> **注意**：`StageEvaluator.recommendation` 字段只输出 `"PASS"` 或 `"REFINE"`，不输出 `"PIVOT"`。PIVOT 策略完全由编排器的分数趋势分析决定。

各阶段的 REFINE/PIVOT 含义：

| 阶段 | REFINE（精化） | PIVOT（转向） |
|------|---------------|--------------|
| script | 保留角色和世界观，强化冲突和喜剧节拍 | 全新创意角度，重建角色关系 |
| characters | 维持设计方向，优化风格一致性和细节 | 切换美术风格流派 |
| storyboard | 保留场景拆分，调整景别和构图 | 重新拆分分镜，改变叙事节奏 |
| video | 调整生成参数（运镜、速度、时长） | 改变合成策略（素材组合方式） |

## 四、质量评估体系

### 4.1 评估流程

```
产物数据 ──► 结构检查（确定性） ──► LLM 深度评估（可选）──► 加权合并 ──► 最终评分
                  │                       │
              总是执行                  当配置可用且
              0-10 分                   结构分 >= 3.0
```

合并策略（`StageEvaluator._merge_results()`）：
- `reason` 含 "需 LLM 评估" 的维度 → **100% 使用 LLM 分**（结构分仅为保底）
- 其他维度 → 结构 40% + LLM 60%
- 两组 issues 去重合并

### 4.2 各阶段验收标准（Stage Contracts）

#### Script（剧本）— 通过门槛 7.0

| 维度 | 权重 | 结构检查逻辑 |
|------|------|-------------|
| 结构完整性 | 2 | `10 - (缺失字段数 × 2.5)` |
| 角色丰富度 | 2 | `(角色字段完成度 / 角色数) × 10 × (实际数 / 期望数)` |
| 剧情连贯性 | 3 | `(场景数 × 2 + 对白数 × 0.5)`，上限 10 |
| 喜剧效果 | 3 | 基础 5.0（需 LLM 评估） |

#### Characters（角色资产）— 通过门槛 7.0

| 维度 | 权重 | 结构检查逻辑 |
|------|------|-------------|
| 完成度 | 4 | `(已填充图片槽 / 总槽数) × 10`，每角色 9 槽（3 视角 + 6 表情） |
| 风格一致性 | 3 | 基础 5.0（需 LLM 评估） |
| 角色辨识度 | 3 | 基础 5.0（需 LLM 评估） |

#### Storyboard（分镜）— 通过门槛 7.0

| 维度 | 权重 | 结构检查逻辑 |
|------|------|-------------|
| 场景覆盖 | 3 | `(已完成面板 / 总面板) × 10` |
| 构图质量 | 3 | `(景别多样性 + 角度多样性) × 1.5`，上限 10 |
| 叙事流畅 | 4 | 基础 5.0（需 LLM 评估） |

#### Video（视频）— 通过门槛 6.0（更宽容，受限于生成模型能力）

| 维度 | 权重 | 结构检查逻辑 |
|------|------|-------------|
| 生成成功 | 5 | 成功且有 URL: 10.0 / 处理中: 3.0 / 错误: 0.0 |
| 时长合理 | 2 | 基础 6.0（需 LLM 评估） |
| 画面质量 | 3 | 基础 6.0（需 LLM 评估） |

### 4.3 LLM 评估提示词格式

```
你是一个视频制作质量评估专家。
请评估以下 {stage} 阶段的产物质量。

## 验收标准
{criteria JSON}

## 产物数据
{artifact JSON (截断到前 8000 字符)}

请输出 JSON:
{
  "scores": [
    {"name": "维度名", "score": 0-10, "reason": "..."}
  ],
  "issues": ["问题1", "问题2"],
  "detail": "总体评价"
}
```

> LLM 参数：`temperature: 0.3`，`max_tokens: 2000`

## 六、并发与安全

### 6.1 并发管线保护

**当前问题**：同一 `project_id` 可重复调用 `POST /start`，会覆盖正在运行的管线状态，两个后台线程竞争写同一记录。

**必须实现**：

```python
def start_pipeline(self, project_id, params, async_run=True):
    # 检查是否已有运行中的管线
    existing = PipelineState.load(project_id)
    if existing and existing.status == PIPELINE_STATUS_RUNNING:
        raise RuntimeError(f"项目 {project_id} 已有运行中的管线")
    # ... 继续创建
```

### 6.2 输入验证

`POST /api/workflow/pipeline/start` 需要验证：

| 字段 | 校验规则 |
|------|---------|
| `project_id` | 非空，匹配 `[a-zA-Z0-9_-]+`，项目必须存在 |
| `genre` | 非空，白名单：喜剧/悬疑/爱情/科幻/古装/都市 |
| `theme` | 非空，长度 10-2000 字符 |
| `characters_count` | 整数，范围 1-10 |
| `episodes_count` | 整数，范围 1-5 |

### 6.3 全管线超时

单阶段轮询超时 600s，但 4 阶段 × 3 重试 × 600s = 最坏情况 **2 小时**。

**需要添加**：全管线 deadline，默认 **30 分钟**。超时后强制终止，保存检查点，标记 `PIPELINE_STATUS_FAILED`。

## 八、状态机映射

### 8.1 Pipeline 状态与 WfProject.status 的同步

**当前问题**：编排器只更新 `PipelineState.status`，**不更新** `WfProject.status`。手动模式前端通过 `WfProject.status` 判断进度，导致托管模式进度在手动模式 UI 中不可见。

**需要实现**：编排器在阶段切换时同步更新 `WfProject.status`：

| Pipeline 事件 | 更新 WfProject.status 为 |
|--------------|------------------------|
| script 开始 | `scripting` |
| script 通过 | `script_parsed` |
| characters 开始 | `designing` |
| characters 通过 | `assets_locked` |
| storyboard 开始 | `storyboarding` |
| video 开始 | `filming` |
| video 通过 | `done` |
| 管线失败 | `compositing_failed` |

实现位置：`WorkflowOrchestrator._run_stage()` 中，在 `mark_stage_running` 和 `mark_stage_passed` 之后调用 `persist_project()` 更新项目状态。

### 8.2 管线完成后衔接手动模式

管线完成后，`WfProject` 处于 `done` 状态。用户可以：
1. 直接下载 MP4
2. 点击「切换到手动模式」→ 将 status 回退到 `assets_locked`，进入手动微调流程

### 10.3 轮询策略

```

  │
  ├─ status === "running" → 继续轮询，更新 UI
  ├─ status === "completed" → 停止轮询，展示报告
  ├─ status === "failed" → 停止轮询，展示错误 + 恢复选项
  └─ status === "paused" → 停止轮询，展示恢复按钮
```

> P1 阶段可升级为 SSE（`/api/workflow/pipeline/events/{project_id}`），消除轮询延迟。

## 十一、错误处理

### 11.1 错误分级

| 级别 | 场景 | 处理方式 |
|------|------|---------|
| **致命** | 关键阶段 (script/video) 3 次重试均失败 | 管线终止，保存检查点 |
| **降级** | 非关键阶段 (characters/storyboard) 失败 | 记录警告，跳过继续 |
| **可恢复** | 网络超时、API 限流 | 自动重试（轮询机制内置） |
| **配置错误** | 引擎模块缺失、API key 无效 | 管线拒绝启动，返回 501 |

### 11.2 产物失败处理

**当前行为**：`_wait_for_completion` 在产物状态为 `error/failed` 时仍返回数据（不是 None），失败产物会被送入评估器打分。

**预期行为**：评估器的结构检查会给失败产物极低分（如 video: 0.0），触发重试。这是 by design，但需要注意评分日志中的失败产物记录。

### 11.3 超时保护

| 环节 | 超时时间 |
|------|---------|
| 单次引擎调用 | 由引擎自行管理 |
| 异步任务轮询 | 600 秒（10 分钟） |
| 后端轮询间隔 | 5 秒 |
| **全管线 deadline** | **30 分钟**（待实现） |
| 前端状态轮询 | 3 秒 |
