# AI 五阶段流水线

> 整合自 docs/01~05 + docs/08，描述从用户输入到最终视频输出的完整生产管线。

---

## 概述：五阶段流水线总览

```
┌─────────┐    ┌─────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 阶段 1  │───▶│ 阶段 2  │───▶│ 阶段 3   │───▶│ 阶段 4   │───▶│ 阶段 5   │
│新建项目 │    │风格定调 │    │剧本解析  │    │资产生成  │    │视频合成  │
│剧本输入 │    │参数配置 │    │结构化拆解│    │角色&场景 │    │分镜&渲染 │
└─────────┘    └─────────┘    └──────────┘    └──────────┘    └──────────┘
  draft       configured     script_parsed   assets_locked      done
```

每个阶段对应一个项目状态流转。用户在手动模式下逐步审核推进；在托管模式下由后端编排器自动循环，附带质量评估门控。

---

## 阶段 1：新建项目与剧本输入

### 用户输入方式

- **项目名称**（可选）：≤50 字符，空则从剧本首行或 AI 摘要自动生成
- **剧本文本**（必填）：200~50,000 字，支持自由格式或结构化格式（故事背景/主要角色/剧情梗概/对白片段）
- **参考图片**（可选）：PNG/JPG/WebP，单张 ≤5MB，最多 5 张

### 快速模板

提供一键填入的剧本模板：霸总甜宠、修仙逆袭、校园暗恋、末世求生，各约 400~500 字。

### 项目创建流程

1. 用户点击「新建项目」填写表单
2. `POST /api/projects` 创建项目目录 `storage/projects/{id}/`
3. 初始化 `project.json`，状态设为 `draft`
4. 参考图片通过 `POST /api/projects/{id}/assets` 上传保存

### 核心数据结构

```typescript
interface WfProject {
  id: string;                         // proj-{日期}-{随机}
  title: string;
  status: string;                     // draft → configured → ...
  script: {
    raw_input: string;
    analysis: ScriptAnalysis | null;
    chat_history: Message[];
  };
  style_config: StyleConfig | null;
  reference_images: string[];         // 参考图片 URL 列表
  characters: Character[];
  scenes: Scene[];
  episodes: Episode[];
  post_production: PostProduction;
}
```

### 校验规则

| 规则 | 处理方式 |
|------|---------|
| 剧本为空 | 禁用提交按钮 + toast 提示 |
| 剧本 < 200 字 | 警告：建议补充更多细节 |
| 剧本 > 50,000 字 | 拒绝：请精简至 50,000 字以内 |
| 图片格式/大小错误 | 前端拦截 |

> 阶段 1 不记录版本历史，因为属于用户原始输入。

## 阶段 2：风格定调与参数配置

### 短片类型

| ID | 名称 | 说明 |
|----|------|------|
| `drama` | 剧情故事片 | 完整叙事结构，起承转合 |
| `music_video` | 音乐概念片 | 配合音乐节奏的视觉叙事 |
| `comic_adapt` | 漫画转视频 | 静态漫画面板转动态视频 |
| `promo` | 产品宣传片 | 品牌/产品动画广告 |
| `edu` | 教育解说片 | 知识讲解动画 |

每种类型附带专属英文提示词和中文结构提示（如：drama → "前3秒出钩子，每10秒一个爽点，结尾致命悬念"）。

### 影片参数

| 参数 | 选项 | 默认值 |
|------|------|--------|
| 故事时长 | 15s / 30s / 60s / 120s / 自定义 | 60s |
| 影片比例 | 16:9 / 9:16 / 1:1 | 9:16 |
| 对白语言 | 中文 / 英文 / 日文 / 韩文 / 无对白 | 中文 |
| 单镜头时长 | 3~8s，步进 1s | 5s |

**自动分镜数计算**：`分镜数 = Math.ceil(总时长 / 单镜头时长)`

### 风格选择

项目已有 **100+ 种风格**，按大类组织：国风、IP风格、日系风格、插画风格、可爱Q版、欧美风格、韩系、立体风格。

每种风格包含 `prompt`（正面提示词）和 `negativePrompt`（反面提示词）。用户通过全屏面板选择风格子类，支持缩略图网格预览。

### 提示词合成

系统自动合成总风格提示词：

```
compiled_style_prompt = 风格子类词 + 类型词 + 比例构图词 + 质量锚定词
compiled_negative_prompt = 风格反面词 + 通用反面词
```

高级用户可展开编辑合成后的提示词，修改后标记 `prompt_manually_edited = true`。

### 数据存储

合成结果保存到 `project.style_config`，项目状态更新为 `configured`。

> 版本历史：重新选择风格/类型或手动编辑提示词时自动存档，最多 10 个版本。

## 阶段 3：剧本解析与结构化拆解

### AI 分析流程

1. **合成解析 Prompt**：剧本文本 + 风格提示词 + 系统指令（含项目参数）
2. **调用 LLM**：流式调用（`claude-sonnet-4-6`，temperature 0.7），前端实时显示进度
3. **解析 JSON**：从 AI 响应中提取 JSON，尝试修复常见格式问题（尾部逗号、单引号、无引号 key）
4. **展示审核**：角色列表卡片 + 场景列表卡片 + 分镜时间线
5. **用户确认**：状态更新为 `script_parsed`

### 输出结构

```
ScriptAnalysis {
  title, genre, style, target_platform, color_palette,
  characters[]:    角色信息 + 三视图提示词 + 表情提示词 + 声线描述
  scenes[]:        场景信息 + 六视图提示词 + 光照/调色描述
  episodes[]:      分集 → 场景列表 → 分镜列表(shots)
                   每个 shot 含景别、运镜、时长、视觉提示词
}
```

**关键约束**：
- 所有图像提示词用英文，其他描述用中文
- 三视图提示词须包含完整外貌/服装/配饰描述
- 分镜拆分标准：场景变化、时间跳转、人物变化、情绪转折、视角改变等
- 一个镜头只讲述一件事

### 用户编辑能力

| 操作 | 说明 |
|------|------|
| 编辑角色/场景/分镜 | 弹窗表单修改 |
| 调整分镜顺序 | 拖拽排序 |
| 单个重新生成 | 发送上下文给 LLM 重新生成 |
| 全部重新生成 | 重新调用完整的 analyzeScript |
| 对话式微调 | 通过侧边栏 ChatPanel 自然语言修改 |

### 评估循环（托管模式）

在托管模式下，剧本解析结果经过 `StageEvaluator` 评估，维度包括：
- 结构完整性（权重 2）、角色丰富度（权重 2）、剧情连贯性（权重 3）、喜剧效果（权重 3）
- 通过门槛：7.0 分

> 版本历史：重新生成、AI 对话微调时自动存档，最多 10 个版本。

## 阶段 4：角色三视图与场景六视图生成

### 资产生成策略

**角色三视图**（每角色 3 张）：
- 正面（front view）、侧面（3/4 side view）、背面（back view）
- 尺寸 1024×1536（竖版全身）
- 提示词来自 `character.three_view_prompts`，自动注入风格关键词

**角色表情图**（主角额外 6 张）：
- neutral、happy、angry、sad、surprised、determined
- 尺寸 512×512（方形）

**场景六视图**（每场景 6 张）：
- 正面、背面、左侧、右侧、俯视、细节特写
- 尺寸 1536×1024（横版全景）

**总计**：每角色最多 9 张（3 视图 + 6 表情），每场景 6 张。

### 并发执行

- 角色和场景的图像生成**并行**执行
- 最大并发数：3 个任务
- 图像生成走中联 MAAS 通道（nano-banana-2），**不走 peiqian.icu**
- 实时进度更新（进度条 + 逐张展示）

### 用户审核与资产锁定

| 操作 | 说明 |
|------|------|
| 通过 | 单张图片标记通过 |
| 重新生成 | 重新调用 API（最多重试 3 次） |
| 上传替代 | 用户上传自己的图片 |
| 编辑提示词 | 修改后重新生成 |

**锁定条件**：所有角色/场景至少正面视图已通过审核。

锁定后资产不可修改（除非解锁回退），提示词固定作为后续分镜参考。项目状态更新为 `assets_locked`。

### 文件存储

```
storage/projects/proj-xxx/assets/
├── char_001_front.png / _side.png / _back.png
├── char_001_expr_neutral.png / _happy.png / ...
├── scene_001_front.png / _back.png / _left.png / _right.png / _top.png / _detail.png
└── ...
```

> 版本历史：重新生成或上传替代时存档（含图片副本），最多 5 个版本。

## 阶段 5：分镜视频生成与最终合成

### Phase A — 分镜图生成

对每个 shot 合成完整的图像提示词（shot 描述 + 景别 + 角色 + 环境 + 风格），调用 nano-banana-2 生成静态分镜参考图。

- 尺寸根据画幅比例选择（16:9 → 1536×1024，9:16 → 1024×1536，1:1 → 1024×1024）
- 并行生成，最多 3 个并发
- 用户逐镜审核：通过 / 重生 / 改提示词

### Phase B — 分镜视频生成

使用 **Seedance 2.0** 视频 API，以分镜图为起始帧生成 3~8s 视频片段。

- 模式：`first_frame`（有分镜图时）或 `text`（无图时）
- 分辨率：720p
- 运镜映射：push_in_slow → push_in, pan_left → pan, static → auto 等
- 逐镜提交任务（间隔 2s 避免限流），轮询直至完成（间隔 5s）

### Phase C — 最终合成

```
视频拼接(FFmpeg) → 转场效果 → 字幕生成(SRT) → AI配音(TTS) → BGM → 音频混缩 → 渲染输出
```

**输出规格**：MP4 (H.264 + AAC)，1080p，24fps

**分阶段流水线**（断点续做）：合成分为 7 个阶段（concat → transition → subtitle → voiceover → bgm → audio_mix → render），跟踪每阶段产物，失败后可从失败阶段重做。

**合成失败处理**：
- 预定义错误分类（FORMAT_MISMATCH、CONCAT_FAILED、SUBTITLE_BURN_FAILED 等）
- 用户可调整参数后重试（字幕模式、配音开关、BGM、质量、转场风格）

> 版本历史：重新生成分镜图/视频或重新合成时存档（仅记录路径引用，不复制视频），最多 5 个版本。

## 托管模式（全自动编排）

### 概述

托管模式允许用户一次提交即完成全链路，无需逐步点击。

| 对比 | 手动模式 | 托管模式 |
|------|---------|---------|
| 人工干预次数 | 10+ | 0（或 1 次最终审核） |
| 端到端耗时 | ~30 min | ~8-15 min |
| 质量保障 | 主观判断 | 自动评分 + 质量门控 |
| 失败恢复 | 手动重试 | 自动 REFINE/PIVOT |

### 四阶段管线

| 阶段 | 引擎 | 生成模型 | 通过门槛 | 关键性 |
|------|------|---------|---------|--------|
| `script` | ScriptEngine | claude-sonnet-4-6 | 7.0 | 关键（失败终止） |
| `characters` | CharacterFactory | nano-banana-2 | 7.0 | 非关键（best effort） |
| `storyboard` | StoryboardGenerator | nano-banana-2 | 7.0 | 非关键（best effort） |
| `video` | VideoComposer | doubao-seedance-2.0 | 6.0 | 关键（失败终止） |

### Builder-Evaluator 协商

每个阶段的执行循环：

```
开始 → 策略决策(首次None/后续REFINE或PIVOT) → BUILD调用引擎 → WAIT轮询完成
    → EVALUATE质量评估 → CHECKPOINT保存 → 通过? → 推进 / 重试
```

**评估体系**：
- 结构检查（确定性，总是执行）+ LLM 深度评估（可选）
- 合并策略：需 LLM 评估的维度 100% 用 LLM 分；其他维度结构 40% + LLM 60%

### REFINE vs PIVOT 决策

由编排器根据分数趋势决定（非评估器返回）：

| 趋势 | 策略 | 含义 |
|------|------|------|
| 大幅进步 (delta > 0.5) | REFINE | 保持方向，精化细节 |
| 有进展 (0 < delta ≤ 0.5) | REFINE | 继续当前路线 |
| 停滞 (delta = 0) | PIVOT | 换方案 |
| 退步 (delta < 0) | PIVOT | 必须换方案 |

### 检查点恢复

- **自动保存时机**：每次 attempt 评估后、阶段通过时、管线完成时
- **数据内容**：pipeline_id、当前阶段、各阶段状态/评分历史/策略/产物ID
- **断点续传**：`POST /api/workflow/pipeline/resume` 加载最新检查点继续执行
- **清理策略**：每项目最多 5 个检查点

### 并发保护与超时

- 同一项目不可重复启动管线（已有 running 状态时拒绝）
- 单阶段轮询超时：600s
- 全管线 deadline：30 分钟（待实现）

## 重试策略说明

所有阶段统一的重试策略，**最多 3 次尝试**：

| 尝试 | 策略 | 说明 |
|------|------|------|
| 第 1 次 | 初始尝试 | 正常执行 |
| 第 2 次 | REFINE | 保持当前方向，根据评估反馈精化输出 |
| 第 3 次 | PIVOT | 如果分数仍未达标，切换到完全不同的方案 |

**失败处理**：
- **关键阶段**（script/video）：3 次均失败 → 管线终止，保存检查点
- **非关键阶段**（characters/storyboard）：3 次均失败 → 记录警告，跳过继续

**可恢复错误**（网络超时、API 限流）：在轮询机制内自动重试，不计入尝试次数。
