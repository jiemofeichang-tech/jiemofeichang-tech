## 1. 无效任务 ID 会误打上游接口

### 现象

访问 `GET /api/tasks/{fake_id}` 时，本地明明不存在该任务，后端仍会继续请求上游任务接口。这样会把本地应返回的 `404` 变成上游 `502` 或 SSL 噪音。

### 根因

`server.py` 的任务查询路由对 task id 只做了“安全字符”校验，没有先判断：

1. 该任务是否已经存在于本地缓存
2. 该 id 是否符合上游任务 id 的实际格式

### 修复

已在后端增加两层保护：

- 本地已缓存且处于终态的任务直接返回，不再请求上游
- 明显不符合上游格式的 task id 直接返回 `404`

同时 `POST /api/library/save` 也增加了同样的拦截，避免对无效 id 发起无意义的上游请求。

### 验证

- 真实登录会话下，`GET /api/tasks/not-a-real-task-id` 返回 `404`
- 真实登录会话下，`GET /api/tasks/cgt-20260328005438-54lwk` 返回 `200`
- 新增回归测试 `tests/test_task_routes.py`

## 2. Windows 后台启动前端时不能直接用 `npm`

### 现象

在 PowerShell 中使用 `Start-Process npm -ArgumentList 'run','dev'` 启动前端会报：

`%1 is not a valid Win32 application`

### 根因

Windows 下 `npm` 实际应通过 `npm.cmd` 启动，`Start-Process` 不能直接把 `npm` 当作可执行文件处理。

### 处理方式

改为使用：

```powershell
Start-Process 'D:\WorkTools\Nodejs\npm.cmd' -ArgumentList 'run','dev'
```

该问题属于本机启动方式问题，不是仓库代码缺陷。

## 3. 这次实际是怎么把项目启动起来的

这次没有新增代码修复；能够稳定启动，关键是把“后台启动前端”的方式改对，并按顺序做健康检查。

### 实际步骤

1. 先确认 `npm.cmd` 的真实路径，而不是直接调用 `npm`

```powershell
where.exe npm.cmd
```

2. 后端直接启动 Python 服务

```powershell
python server.py
```

3. 如果需要在后台启动前端，必须显式调用 `npm.cmd`

```powershell
Start-Process 'D:\WorkTools\Nodejs\npm.cmd' `
  -ArgumentList 'run','dev' `
  -WorkingDirectory 'D:\AiProject\AiComedyDrama'
```

4. 启动后立即做两个健康检查

```powershell
Invoke-WebRequest -UseBasicParsing 'http://127.0.0.1:8787/api/health'
Invoke-WebRequest -UseBasicParsing 'http://localhost:3001/login'
```

### 为什么这次能成功

- 后端本身这次没有新增启动故障
- 前端报错点仍然是 Windows 下 `Start-Process npm` 的老问题
- 改为 `npm.cmd` 后，Next.js dev server 能正常监听 `3001`
- 再用健康检查确认前后端都已真正就绪，而不是只看进程是否存在

### 适用场景

这套方法特别适用于通过脚本、自动化工具、Cloud/Codex 代理进程在后台拉起前端的场景。前台直接执行 `npm run dev` 往往不会暴露这个问题，但后台启动经常会踩到。
